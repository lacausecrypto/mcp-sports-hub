## How Providers Work

### The Register Pattern

Every provider file exports a single function:

```typescript
export function register(server: McpServer): void {
  // Register tools on the server
  server.tool("prefix_tool_name", "Description", { /* zod schema */ }, handler);
}
```

The entry point (`src/index.ts`) imports all 18 register functions and calls them in sequence on the same `McpServer` instance:

```typescript
const server = new McpServer({ name: "sports-hub", version: "1.0.0" });

registerEspn(server);
registerNhl(server);
registerMlbStats(server);
// ... 15 more providers ...

const transport = new StdioServerTransport();
await server.connect(transport);
```

This means all 205 tools are registered on a single MCP server and exposed over a single stdio connection.

### Tool Registration

Each tool is registered with four arguments:

1. **Name** -- Prefixed with the provider name (e.g., `espn_get_scoreboard`)
2. **Description** -- A concise human-readable description
3. **Input schema** -- A Zod schema object defining the parameters
4. **Handler** -- An async function that calls the API and returns results

Example:

```typescript
server.tool(
  "nhl_get_standings",
  "Get NHL standings. Optionally filter by date or season.",
  {
    date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
    season: z.string().regex(/^\d{8}$/).optional(),
  },
  async ({ date, season }) => {
    // ... fetch data and return toolResult(data)
  },
);
```

## Shared HTTP Module

`src/shared/http.ts` provides four core utilities used by all providers:

### `fetchJson(url, options?)`

Fetches a URL and returns parsed JSON. Handles error status codes by throwing descriptive errors that include the HTTP status and response body. Sets standard headers (`Accept: application/json`, `User-Agent: mcp-sports-hub/1.0.0`) and merges in any provider-specific headers.

### `buildUrl(base, params?)`

Constructs a URL with query parameters, automatically skipping any that are `undefined`, `null`, or empty string. This avoids the need for manual parameter filtering in each provider.

### `toolResult(data)`

Wraps data as a standard MCP tool response:

```typescript
{ content: [{ type: "text", text: JSON.stringify(data, null, 2) }] }
```

### `errorResult(message)`

Wraps an error message as a standard MCP error response:

```typescript
{ content: [{ type: "text", text: `Error: ${message}` }], isError: true }
```

## Design Decisions

### Prefixed Tool Names

All tools use the pattern `{provider}_{action}` (e.g., `espn_get_scoreboard`, `nhl_get_standings`). This prevents name collisions between providers that offer similar functionality (many providers have "get_standings" or "get_fixtures") and makes it clear which API is being called.

### Lazy Key Checking

API keys are checked inside each tool handler, not at module import time or during registration. This means:

- The server always starts successfully, regardless of which keys are configured.
- All 205 tools are always registered and visible to the client.
- Calling a tool without its required key returns a clear error like `"API_FOOTBALL_KEY env var is required"`.

This is intentional. It allows users to configure keys incrementally and discover what is available before committing to registration on every API.

### Single Server, Stdio Transport

All providers run in a single process on a single stdio transport. This keeps configuration simple (one entry in Claude Desktop config) and avoids the overhead of managing multiple server processes. The tradeoff is that all 205 tools appear in the tool list, but the prefixed naming makes them easy to filter.

### No Caching Layer

The server does not cache API responses. Each tool call makes a fresh HTTP request. Caching would add complexity and stale-data risks. Callers (AI assistants) can manage their own caching at the conversation level.

### Zod for Input Validation

All tool parameters are defined using Zod schemas. The MCP SDK automatically validates inputs against these schemas before calling the handler. This provides type safety and clear error messages for invalid inputs.

## Dependencies

The server has only two runtime dependencies:

- `@modelcontextprotocol/sdk` -- The official MCP SDK for tool registration and stdio transport
- `zod` -- Schema validation for tool parameters

All HTTP calls use the built-in `fetch` API (Node.js 18+). No additional HTTP libraries are needed.

## How to Add a New Provider

1. **Create the provider file.** Add `src/providers/my-provider.ts`.

2. **Export a register function.**

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";
import { fetchJson, buildUrl, toolResult, errorResult } from "../shared/http.js";

export function register(server: McpServer): void {
  const API_KEY = process.env.MY_PROVIDER_API_KEY;

  server.tool(
    "myprovider_get_data",
    "Description of what this tool does",
    {
      param1: z.string().describe("Parameter description"),
      param2: z.number().optional().describe("Optional parameter"),
    },
    async ({ param1, param2 }) => {
      if (!API_KEY) return errorResult("MY_PROVIDER_API_KEY env var is required");
      try {
        const url = buildUrl("https://api.example.com/data", { param1, param2 });
        const data = await fetchJson(url, {
          headers: { Authorization: `Bearer ${API_KEY}` },
        });
        return toolResult(data);
      } catch (err) {
        return errorResult(err instanceof Error ? err.message : String(err));
      }
    },
  );
}
```

3. **Prefix all tool names** with your provider name (e.g., `myprovider_`).

4. **Check the API key inside the handler**, not at the top level.

5. **Register in the entry point.** Edit `src/index.ts`:

```typescript
import { register as registerMyProvider } from "./providers/my-provider.js";
// ...
registerMyProvider(server);
```

6. **Build and test.**

```bash
npm run build
```

### Provider Checklist

- [ ] All tool names prefixed with `{provider}_`
- [ ] API key checked inside handler (not at import time)
- [ ] Uses `fetchJson` / `buildUrl` from shared HTTP module
- [ ] Returns `toolResult(data)` on success, `errorResult(message)` on failure
- [ ] All parameters have `.describe()` annotations
- [ ] Optional parameters use `.optional()`
- [ ] Registered in `src/index.ts`
